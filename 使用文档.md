# multiple-namespace 使用文档

## 1. 目标

这个项目的目标很明确：

- 服务端可以把同一个 Dubbo 服务注册到多个 Nacos namespace
- 消费端可以自己选择：
  - 继续使用原生 `nacos://`
  - 或使用多空间协议 `multins-nacos://`

核心边界：

- `nacos://` 按宿主项目原生 Dubbo / Nacos 逻辑执行
- `multins-nacos://` 才启用多空间注册与多空间消费能力

## 2. 模块说明

项目拆成两个模块：

- `multiple-namespace-dubbo2`
  - 适用于 Dubbo `2.7.x`
- `multiple-namespace-dubbo3`
  - 适用于 Dubbo `3.3.x`

宿主项目必须按自己的 Dubbo 主版本选择对应模块。

## 3. 能力范围

当前支持的能力：

- Provider 多 namespace 注册
- Consumer 多 namespace 订阅
- 主空间优先
- 备用空间 fallback
- Nacos 下线实例在多空间消费链中不再参与选择

当前不接管的能力：

- 原生 `nacos://` 的消费策略
- Dubbo 3 application-level discovery 的默认行为
- 宿主项目自己的 metadata center / config center 决策

## 4. Maven 依赖

### 4.1 Dubbo 2 项目

```xml
<dependency>
    <groupId>com.huanxin</groupId>
    <artifactId>multiple-namespace-dubbo2</artifactId>
    <version>0.0.1-SNAPSHOT</version>
</dependency>
```

### 4.2 Dubbo 3 项目

```xml
<dependency>
    <groupId>com.huanxin</groupId>
    <artifactId>multiple-namespace-dubbo3</artifactId>
    <version>0.0.1-SNAPSHOT</version>
</dependency>
```

说明：

- 这两个模块里的 `dubbo`、`nacos-client`、`slf4j-api` 都是 `provided`
- 不会主动覆盖宿主项目自己的 Dubbo 版本

## 5. 服务端使用方式

### 5.1 多空间注册

服务端如果要注册到多个 namespace，请使用 `multins-nacos://`：

```yaml
dubbo:
  registry:
    address: multins-nacos://127.0.0.1:8848?username=nacos&password=nacos
    register-mode: all
    use-as-config-center: false
    use-as-metadata-center: true
    parameters:
      namespace: dev,test
```

上面这段的含义：

- 实例会注册到 `dev`
- 实例也会注册到 `test`
- 主空间默认取第一个，也就是 `dev`

### 5.2 只注册到单空间

如果服务端不需要多空间注册，继续用原生 `nacos://` 即可：

```yaml
dubbo:
  registry:
    address: nacos://127.0.0.1:8848?username=nacos&password=nacos&namespace=dev
```

## 6. 消费端使用方式

## 6.1 使用原生协议

如果消费端要继续按原生逻辑消费，请直接使用 `nacos://`：

```yaml
dubbo:
  registry:
    address: nacos://127.0.0.1:8848?username=nacos&password=nacos
    register: false
    subscribe: true
    use-as-config-center: false
    use-as-metadata-center: true
    parameters:
      namespace: dev
```

说明：

- 这里就是原生 Dubbo / Nacos 消费
- 不启用多空间聚合
- 只消费 `dev` 里的实例

## 6.2 使用多空间协议

如果消费端需要主空间优先和 fallback，请使用 `multins-nacos://`：

```yaml
dubbo:
  registry:
    address: multins-nacos://127.0.0.1:8848?username=nacos&password=nacos
    register: false
    subscribe: true
    use-as-config-center: false
    use-as-metadata-center: true
    parameters:
      namespace: dev,test
      namespaceSelectMode: primary-first
      namespaceFallback: true
      healthyOnly: true
```

这时行为是：

- 主空间：`dev`
- 备用空间：`test`
- 先查 `dev`
- `dev` 没有可用实例时，再查 `test`

## 6.3 只覆盖消费空间

如果 registry 想保留默认配置，但消费端只想覆盖消费空间，可以这样写：

```yaml
dubbo:
  consumer:
    parameters:
      namespace: dev,test
```

效果：

- 只影响消费发现链
- 不影响 Provider 注册链

## 7. 默认规则

- `primaryNamespace` 不配置时，默认取 `namespace` 的第一个
- `registerNamespaces` 不配置时，默认等于 `namespace`
- `subscribeNamespaces` 不配置时，默认等于 `namespace`

例如：

```yaml
parameters:
  namespace: dev,test
```

默认等价于：

- `primaryNamespace=dev`
- `registerNamespaces=[dev,test]`
- `subscribeNamespaces=[dev,test]`

## 8. 元数据说明

当前版本为了兼容 `multins-nacos://` 同时作为 metadata center 的场景，提供了 `MetadataReportFactory` 适配。

规则是：

- 如果宿主项目把 metadata center 也配置成 `multins-nacos://`
- metadata 会收敛到主空间
- 主空间默认取 `namespace` 第一个

例如：

```yaml
parameters:
  namespace: dev,test
```

则 metadata 默认落到：

- `dev`

## 9. 原生协议和多空间协议的区别

### 9.1 `nacos://`

- 完全原生
- 不做多空间聚合
- 不做 fallback
- 是否能消费，取决于宿主项目原生 Dubbo 配置

### 9.2 `multins-nacos://`

- Provider 可多空间注册
- Consumer 可多空间消费
- 支持主空间优先
- 支持 fallback
- 会过滤 Nacos 中已下线 `enabled=false` 的实例

## 10. Nacos 下线行为说明

在 `multins-nacos://` 下：

- 如果某个 namespace 的实例被 Nacos 标记为下线
- 该实例不会再参与多空间消费选择

注意：

- `Nacos 下线` 不等于立刻断开已有 Dubbo 连接
- 它只影响后续注册发现结果

## 11. 常见问题

### 11.1 启动时报 `No such extension MetadataReportFactory by name multins-nacos`

原因：

- 本地 Maven 仓库里仍然是旧 jar

处理：

1. 删除本地仓库里旧版本目录
2. 在本项目根目录重新执行 `mvn clean install`
3. 重新刷新宿主项目依赖

### 11.2 消费端看不到服务

先确认：

1. Provider 和 Consumer 使用的 Dubbo 主版本是否一致
2. Provider 引用的是不是正确模块
3. Consumer 当前使用的是 `nacos://` 还是 `multins-nacos://`
4. 当前 namespace 是否一致
5. Nacos 中实例是否真的存在且未下线

### 11.3 点击 dev 下线后为什么还能调

先看消费端日志：

- 是否收到 `namespace=dev` 的变更事件
- 是否打印了 `事件实例已被多空间消费链过滤`
- 是否打印了 `当前返回给 Dubbo 的实例明细：[]`

如果变更事件没到，就是 Nacos 没推到消费端。

如果事件到了但还能调，通常是旧 invoker 还没被替换；当前版本已经按原始订阅 URL 回发 `empty://` 通知来清空旧地址。

## 12. 推荐接入方式

### 场景一：只想服务端多空间注册，消费端原生消费

- Provider：`multins-nacos://`
- Consumer：`nacos://`

### 场景二：服务端多空间注册，消费端也要多空间消费

- Provider：`multins-nacos://`
- Consumer：`multins-nacos://`

### 场景三：完全不需要多空间

- Provider：`nacos://`
- Consumer：`nacos://`
